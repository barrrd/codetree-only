## ⚠️ 구현 시 주의해야 할 핵심 포인트

### 1) '올바른 괄호 문자열'의 원리를 출입 시스템에 투영해야 함
이 문제는 괄호 쌍 찾기 알고리즘의 변형입니다. 사무실 출입 기록이 정상적이기 위해서는 단순히 총 개수가 맞는 것을 넘어, 모든 시점에서 논리적 모순이 없어야 합니다.

* **핵심 조건:**
    1. **중간 검증:** 어느 시점에서도 (현재까지 사무실에 들어온 사람 수) $\ge$ (현재까지 사무실에서 나간 사람 수)여야 합니다. 들어오지 않은 사람이 먼저 나갈 수는 없기 때문입니다.
    2. **최종 검증:** 모든 기록이 끝났을 때, (총 들어온 사람 수) == (총 나간 사람 수)여야 합니다. 퇴근하지 않고 사무실에 남은 사람이 있으면 안 됩니다.
* **➡️ 해결 방법:** `(`를 $+1$, `)`를 $-1$로 치환했을 때, 누적합이 한 번도 음수가 되지 않아야 하며 최종 합은 $0$이어야 합니다.

---

### 2) 모든 경우의 수(Brute Force)를 따지면 반드시 시간 초과 발생
물음표(`?`)가 최대 $N$개까지 주어질 수 있으므로, 각 `?`를 `(` 또는 `)`로 채우는 $2^N$가지 경우를 모두 확인하는 방식은 불가능합니다.

* **오류 예시:** 재귀 함수나 백트래킹을 이용해 모든 `?` 조합을 만들어보고 올바른 괄호인지 체크하는 방식.
* **핵심:** 이 문제는 "가능한 경우가 단 하나라도 존재하는가?"를 묻고 있습니다. 따라서 **가장 유리한 시나리오** 하나만 만들어 검증하는 **그리디(Greedy)** 접근이 필요합니다.

---

### 3) 그리디 전략: "일단 들어오게(`(`) 하는 것이 생존에 유리함"
중간에 인원수가 음수가 되는 상황(나간 사람이 더 많은 상황)을 피하기 위한 최선의 전략은 무엇일까요?

* **핵심:** 사람이 최대한 빨리, 많이 들어와 있어야 나중에 `)`가 나와도 버틸 수 있습니다. 즉, **`(`는 기록의 앞쪽에 나타날수록, `)`는 뒤쪽에 나타날수록** 전체 시스템이 정상일 확률이 높아집니다.
* **➡️ 해결 방법:**
    1. 전체 기록의 길이 $N$이 홀수이면 즉시 `No`.
    2. 최종적으로 필요한 `(`의 총 개수는 $N/2$개입니다.
    3. 이미 확정된 `(`의 개수를 뺀 나머지 **할당량(`need_in`)**을 계산합니다.
    4. 앞에서부터 `?`를 만날 때마다, **`need_in`이 남아있다면 무조건 `(`로 채웁니다.**
    5. `need_in`을 다 썼다면 그 이후의 모든 `?`는 `)`로 채웁니다.

---

### 4) 미래의 확정된 기록을 위한 "자리 비워두기"
단순히 `curr_in < N/2` 조건만 보고 `?`를 `(`로 바꾸면 안 됩니다. 기록의 뒷부분에 이미 확정된 `(`가 나올 수 있기 때문입니다.

* **오류 예시:** `?`를 만났을 때 현재까지의 `curr_in`이 $N/2$보다 작다고 해서 무턱대고 `(`로 바꾸는 경우. 뒤에서 나올 확정된 `(` 때문에 전체 `(` 개수가 $N/2$를 초과하게 됩니다.
* **핵심:** `?`가 쓸 수 있는 `(`의 개수는 **(전체 할당량 $N/2$) - (이미 문자열에 존재하는 '(' 의 개수)**로 엄격하게 제한해야 합니다.
* **➡️ 해결 방법:**
    ```python
    # 물음표가 가져갈 수 있는 '('의 기회 횟수 미리 계산
    need_in = (n // 2) - records.count('(')
    
    # 루프 내부에서 ? 를 만났을 때
    if need_in > 0:
        curr_in += 1
        need_in -= 1  # 기회 소모
    else:
        curr_out += 1
    ```

---

### 5) 완성된 "최선의 시나리오"를 단 한 번 검증
가장 유리하게 구성한 시나리오가 올바른 괄호 문자열의 규칙을 만족하는지 마지막으로 확인합니다.

* **➡️ 해결 방법:**
    * 구성된 문자열을 처음부터 순회하며 실시간 인원수를 계산합니다.
    * 중간에 `curr_in < curr_out`이 되는 순간이 단 한 번이라도 있다면 즉시 `No`.
    * 루프 종료 후 `(curr_in, curr_out)`이 정확히 `(N/2, N/2)`가 아니면 `No`.
    * 위 필터를 모두 통과하면 `Yes`.

---

## 풀이 핵심 요약

이 문제는 **그리디(Greedy)** 알고리즘을 통해 "가장 정상일 가능성이 높은 문자열"을 단 하나 생성하여 판별하는 문제입니다.

1. $N$이 짝수인지 확인하고, 필요한 `(`의 잔여 할당량(`need_in`)을 계산한다.
2. 앞에서부터 등장하는 `?`에 `need_in`을 우선 배분하여 `(`로 변환한다.
3. 변환된 문자열이 **모든 시점에서 `in >= out`**이고 **최종적으로 `in == out`**인지 확인한다.
4. 이 "최선의 시나리오"마저 실패한다면, 어떤 조합으로도 정상 기록을 만들 수 없다.

## 사용한 알고리즘 및 문법

- **Greedy (탐욕법):** 최적의 해를 찾기 위해 매 순간 유리한 선택(`(` 선배치)을 수행
- **문자열 전처리:** `count()` 함수를 이용한 빈도 분석
- **누적합 (Prefix Sum):** 실시간 인원수 변동 및 모순 추적
